# Рекомендации по оптимизации Shadow Stack Widget для macOS M1 (8GB RAM)

Текущая архитектура (Playwright, Ollama, React, Telegram Bot) потребляет значительное количество ресурсов. При работе на устройстве с 8 ГБ объединенной памяти (Unified Memory) критически важно управлять жизненным циклом тяжелых процессов.

## 1. Защита от утечек памяти в PM2 (`ecosystem.config.cjs`)
В текущем файле конфигурации PM2 отсутствуют лимиты потребления памяти.
**Решение:**
Добавьте параметр `max_memory_restart` ко всем Node.js процессам. Это заставит PM2 автоматически перезапускать сервисы (`shadow-api`, `shadow-bot`), если они начнут потреблять слишком много ОЗУ.
Пример:
```javascript
{
  name: 'shadow-api',
  script: './server/index.js',
  max_memory_restart: '300M', // Лимит 300 МБ
  env: env
}
```

## 2. Глобальное ограничение удержания моделей Ollama
В конфигурации `openclaw.config.json` параметр `keep_alive: 0` уже настроен для выгрузки моделей. Однако, прямые вызовы к Ollama (минуя OpenClaw) могут оставлять модель в VRAM (~2ГБ для qwen2.5-coder:3b).
**Решение:**
Добавьте в глобальные переменные среды (`~/.zshrc` или `.env`) настройку:
```env
OLLAMA_KEEP_ALIVE=0
```
Это обеспечит немедленную выгрузку любых моделей из оперативной памяти после завершения их генерации.

## 3. Автоматизация запуска Chrome CDP в `tmux-shadow.sh`
Shadow Router (`server/shadow-router.cjs`) опирается на запущенный инстанс Chrome в режиме отладки (порт 9222). Сейчас скрипт `tmux-shadow.sh` не запускает браузер автоматически, что может приводить к ошибкам `ECONNREFUSED`.
**Решение:**
Добавьте запуск браузера в начало скрипта `scripts/tmux-shadow.sh`:
```bash
# Запуск Chrome для Shadow Router в фоновом режиме
open -a "Google Chrome" --args --remote-debugging-port=9222 &
sleep 2
```

## 4. Настройка Telegram Bot Token
Ошибки `401 Unauthorized` в логах бота указывают на недействительный или пустой токен.
**Решение:**
Получите новый токен в Telegram через `@BotFather` и укажите его в корне проекта в файле `.env`:
```env
TELEGRAM_BOT_TOKEN="1234567890:AAExxxxx..."
```
Без этого токена процесс `shadow-bot` будет циклически падать.

## 5. Избегайте Vercel локально
Деплой проекта на Vercel (через CLI-пакеты вроде `vercel`) тянет множество тяжелых зависимостей (например, `undici`), содержащих десятки уязвимостей. 
**Решение:**
Мы уже удалили `vercel` из зависимостей. Поддерживайте это состояние и запускайте Health Dashboard исключительно локально на порту `:5176`.

# Часть 2. Среднесрочные и Долгосрочные оптимизации (Адаптация плана Manus)

## 6. Кэширование запросов к LLM (Prompt Cache)
Ollama (например, `qwen2.5-coder:3b`) загружается в VRAM (~2ГБ) и сильно нагружает CPU/GPU M1 даже для одинаковых рутинных запросов.
**Решение:**
Поддерживать In-Memory кэш (LRU Cache или простой `Map`) на уровне `server/lib/ai-sdk.cjs`. Если текст запроса (prompt) полностью совпадает с недавним, система должна моментально вернуть ответ из кэша, сэкономив 90% времени и ресурс батареи.

## 7. Улучшение UX (Fallback Messages в боте)
Когда свободного ОЗУ меньше 400МБ, локальный `shadow-router` отключает `Playwright` и перенаправляет запросы через API. Пользователь об этом не знает.
**Решение:**
В `bot/opencode-telegram-bridge.cjs` при получении статуса `503 LOW_RAM` бот должен отправлять сервисное уведомление: `⚠️ RAM < 400МБ. Браузерная модель временно недоступна, запрос отправлен через резервный API`. Это создаст прозрачность системы для владельца.

## 8. Локальный трекинг ошибок (Silent Error Tracking)
Вместо отправки логов в облачные сервисы (Sentry), что расходует сеть и может нарушить приватность, ошибки нужно агрегировать локально.
**Решение:**
Добавить обработчики `uncaughtException` и `unhandledRejection`, сохраняющие стек-трейсы в `data/crash.log` для последующего анализа через Health Dashboard.

## 9. Обучение на отзывах (Feedback Loop)
Интеграция с локальной векторной базой `ChromaDB`.
**Решение:**
Сохранять удачные решения ИИ (при подтверждении от пользователя) через скрипт `scripts/index_knowledge.py` для улучшения будущего контекста (RAG).
